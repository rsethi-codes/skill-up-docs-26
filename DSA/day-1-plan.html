🚫 The Most Expensive DSA Mistake

Using list.pop(0) for queue operations. This is O(n) — not O(1). Every element shifts left. In a 50,000-element list, this is a performance disaster. Always use collections.deque for queues. This distinction alone has failed candidates in interviews.

0.3 Big-O — The Language of DSA

Big-O describes how your algorithm scales as input grows. It's not about actual speed — it's about the shape of growth. You must state Big-O after every solution you write, every time. It becomes automatic if you practice it from day one.

🎬 WATCH THIS FIRST — Big O Notation in 100 Seconds (Fireship)
Big-O Growth Rates — From Best to Worst
  O(1)       → Constant    — dict lookup, array index, push/pop stack
  O(log n)   → Logarithmic — binary search, heap push/pop
  O(n)       → Linear      — single loop, most array traversals
  O(n log n) → Linearithmic — sorting (Timsort, merge sort, heap sort)
  O(n²)      → Quadratic   — nested loops — AVOID in interviews
  O(2ⁿ)      → Exponential — recursion without memo (subsets, backtracking)
  O(n!)      → Factorial   — generating all permutations

  For n = 1,000:
  O(1)       →         1 operations
  O(log n)   →        10 operations
  O(n)       →     1,000 operations
  O(n log n) →    10,000 operations
  O(n²)      → 1,000,000 operations  ← often too slow
💡 The Interview Rule for Big-O

After every solution, immediately say: "Time complexity is O(n) because we iterate once. Space complexity is O(n) for the hash map." Do this even in practice — solo. It trains the habit. Interviewers mark down candidates who can't immediately state complexity. The answer doesn't have to be perfect; the instinct to analyze it does.

Amortized O(1) — Why list.append() is "O(1)"

Python lists double their capacity when full. So most appends are O(1), but occasionally one is O(n) (copying to new memory). Averaged across many operations, this is O(1) — that's what "amortized" means. This is why you'll see "O(1) amortized" in Python docs.

Arrays Deep Dive

Arrays are the bedrock of DSA. Everything else — stacks, queues, graphs — is built on or adjacent to arrays. Understanding them at the memory level helps you reason about why certain operations are fast or slow, and which patterns exploit that structure.

🎬 Arrays in Python — NeetCode (10 min, highly recommended)

1.1 How Arrays Work in Memory

🧠 The Mental Model

An array is a contiguous block of memory. Every element sits right next to the previous one. This is why index access is O(1) — you just do math: base_address + index * element_size. The computer can jump directly to any element without searching.

Array Memory Layout — arr = [10, 20, 30, 40, 50]
  Memory Address:   0x100  0x104  0x108  0x10C  0x110
                   ┌──────┬──────┬──────┬──────┬──────┐
  arr =            │  10  │  20  │  30  │  40  │  50  │
                   └──────┴──────┴──────┴──────┴──────┘
  Index:             [0]    [1]    [2]    [3]    [4]

  arr[2] = 0x100 + 2 * 4 bytes = 0x108 → value = 30
  This calculation is O(1) — no searching required.

  arr[-1] = arr[4] = 50  ← Python handles negative indices for you
  arr[-2] = arr[3] = 40

This contiguous layout means insertions and deletions in the middle are expensive — O(n) — because every element after the insertion point must shift. This is the fundamental tradeoff of arrays vs linked lists.

⚠️ Python List ≠ C Array

Python's list is actually a dynamic array — it auto-resizes. When full, Python allocates a new, larger block and copies everything over. This is why append() is amortized O(1) but can occasionally be O(n). For true fixed-size arrays, use array.array or numpy.array — but for DSA interviews, list is always sufficient.

1.3 Prefix Sums — Turning O(n) Range Queries into O(1)

🧠 The Mental Model

A prefix sum array stores cumulative sums. Once built, you can answer "what's the sum of elements from index L to R?" in O(1) using subtraction — instead of looping through the subarray each time. It's a classic precomputation strategy: spend O(n) once, save O(n) per query forever.

Prefix Sum — Visual Walkthrough
  Original:  arr   = [3,  1,  4,  1,  5,  9,  2,  6]
  Index:              0   1   2   3   4   5   6   7

  prefix[0] = 0  (empty sum — the sentinel)
  prefix[1] = 0 + 3 = 3
  prefix[2] = 3 + 1 = 4
  prefix[3] = 4 + 4 = 8
  prefix[4] = 8 + 1 = 9
  prefix[5] = 9 + 5 = 14
  prefix[6] = 14 + 9 = 23
  prefix[7] = 23 + 2 = 25
  prefix[8] = 25 + 6 = 31

  Sum from index 2 to 5 (inclusive):
  = prefix[6] - prefix[2]
  = 23 - 4
  = 19   ✓  (4 + 1 + 5 + 9 = 19)

  Formula: sum(L, R) = prefix[R+1] - prefix[L]
python prefix sums — build and query 
# ── BUILD PREFIX SUM ARRAY ──────────────────────────────────────
def build_prefix(arr):
    n = len(arr)
    prefix = [0] * (n + 1)    # extra 0 at front (the sentinel)
    for i in range(n):
        prefix[i + 1] = prefix[i] + arr[i]
    return prefix

# ── RANGE SUM QUERY ─────────────────────────────────────────────
def range_sum(prefix, L, R):
    # Sum of arr[L..R] (both inclusive)
    return prefix[R + 1] - prefix[L]

arr = [3, 1, 4, 1, 5, 9, 2, 6]
prefix = build_prefix(arr)
print(range_sum(prefix, 2, 5))   # 19 = 4+1+5+9
print(range_sum(prefix, 0, 7))   # 31 = sum of entire array

# ── REAL PROBLEM: Subarray Sum Equals K (LC #560) ───────────────
# How many subarrays sum to target k?
# Key insight: if prefix[j] - prefix[i] == k, then arr[i..j-1] sums to k
# Equivalently: prefix[j] - k == prefix[i] for some earlier i
# So: as we build prefix, track "seen prefix sums" in a hash map

def subarray_sum(nums, k):
    count = 0
    curr_sum = 0
    seen = {0: 1}    # prefix sum 0 has been seen once (the empty prefix)
    
    for num in nums:
        curr_sum += num
        # if curr_sum - k was seen before, those prefix sums form valid subarrays
        count += seen.get(curr_sum - k, 0)
        seen[curr_sum] = seen.get(curr_sum, 0) + 1
    
    return count

# Time: O(n), Space: O(n)
# Without prefix sums: O(n²) brute force
TIME O(n) build, O(1) query
SPACE O(n) for prefix array
✅ When to Reach for Prefix Sums

Any time you see the phrase "sum of subarray", "sum of range [L, R]", or "number of subarrays with property X related to sum" — prefix sums are your tool. The Subarray Sum Equals K problem (LC #560) is the canonical example — it uses prefix sums + hash map to go from O(n²) to O(n).

1.4 Kadane's Algorithm — Maximum Subarray

Kadane's algorithm finds the contiguous subarray with the largest sum in O(n). It's the classic example of dynamic programming thinking applied to arrays — the insight is that at each position, you make one decision: extend the current subarray or start fresh.

Kadane's — Visual Trace for [-2, 1, -3, 4, -1, 2, 1, -5, 4]
  At each index, decide: max(num, curr_sum + num)
  "Should I start fresh with just num, or extend the current window?"

  Index:   0     1     2     3     4     5     6     7     8
  num:    -2     1    -3     4    -1     2     1    -5     4
  curr:   -2     1    -2     4     3     5     6     1     5
  max:    -2     1     1     4     4     5     6     6     6

  curr at index 1: max(1, -2+1) = max(1, -1) = 1  ← start fresh!
  curr at index 3: max(4, -2+4) = max(4, 2) = 4   ← start fresh again!
  curr at index 5: max(2, 3+2) = max(2, 5) = 5    ← extend (5 > 2)

  Answer: 6  (subarray [4, -1, 2, 1])
python kadane's algorithm — complete with explanation 
# LC #53 — Maximum Subarray
def max_subarray(nums):
    max_sum = nums[0]    # global maximum (answer)
    curr_sum = nums[0]   # current window sum
    
    for i in range(1, len(nums)):
        num = nums[i]
        # DECISION: start fresh (num alone) OR extend current window
        curr_sum = max(num, curr_sum + num)
        max_sum = max(max_sum, curr_sum)
    
    return max_sum

# One-liner version (same logic)
def max_subarray_v2(nums):
    curr = best = nums[0]
    for n in nums[1:]:
        curr = max(n, curr + n)
        best = max(best, curr)
    return best

# ── Extension: return the actual subarray (not just sum) ─────────
def max_subarray_with_indices(nums):
    max_sum = curr_sum = nums[0]
    start = end = temp_start = 0
    
    for i in range(1, len(nums)):
        if nums[i] > curr_sum + nums[i]:    # starting fresh is better
            curr_sum = nums[i]
            temp_start = i
        else:
            curr_sum += nums[i]
        
        if curr_sum > max_sum:
            max_sum = curr_sum
            start = temp_start
            end = i
    
    return max_sum, nums[start:end+1]

print(max_subarray([-2,1,-3,4,-1,2,1,-5,4]))  # 6
TIME O(n)
SPACE O(1) — only 2 variables!
💡 The DP Insight in Kadane's

Kadane's is actually 1D dynamic programming in disguise. curr_sum represents: "the maximum subarray sum ending exactly at this index." For each index, we choose: either this element starts a new subarray, or it extends the one ending at the previous index — whichever is larger. This is the same "choice at each step" pattern you'll see in House Robber, Coin Change, and many other DP problems in Week 10.

Section 02

Hashing Deep Dive

Hashing is the superpower of DSA — it converts O(n) lookups into O(1). Almost every "optimize from O(n²) to O(n)" trick involves introducing a hash map. Understand hashing deeply once, and you'll see it everywhere forever.

🎬 Hashing Deep Dive — NeetCode (essential watch)

2.1 How Hash Maps Work Internally

🧠 The Mental Model

A hash map is an array under the hood. To store key → value, it computes hash(key) % array_size to find the bucket index, then stores there. Lookup is O(1) because you just hash the key and jump directly to the bucket — no searching. Collisions (two keys mapping to the same bucket) are handled via chaining (linked list at each bucket) or open addressing.

Hash Map — How dict Works Under the Hood
  Storing: {"apple": 1, "banana": 2, "cherry": 3}

  hash("apple")  % 8 = 5  → bucket[5] = ("apple", 1)
  hash("banana") % 8 = 2  → bucket[2] = ("banana", 2)
  hash("cherry") % 8 = 7  → bucket[7] = ("cherry", 3)

  Backing array:
  Index: [ 0  ][ 1  ][ 2          ][ 3 ][ 4 ][ 5         ][ 6 ][ 7          ]
         [None][None]["banana"→ 2 ][None][None]["apple"→ 1][None]["cherry"→ 3]

  Lookup "apple": hash("apple") % 8 = 5 → return bucket[5].value = 1
  One step. No loop. O(1).

  Collision: if hash("mango") % 8 = 5 too → chaining:
  bucket[5] = [("apple", 1) → ("mango", 4)]
  Worst case (all collide): O(n) — but in practice, avg is O(1)

This is why dict and set have O(1) average lookups. In the worst case (pathological hash collisions), they degrade to O(n) — but Python's hash function is designed to make this essentially impossible with normal usage.

⚠️ What Can Be a Key? (Hashability)

Only immutable objects can be dict keys or set elements, because their hash must never change. str, int, float, tuple ✅. list, dict, set ❌ — they're mutable, so they can't be hashed. This is why you'll often see tuple(sorted(word)) as a hashable key for anagram grouping.

# ── DEFAULTDICT ───────────────────────────────────────────────────
# defaultdict(type_factory) — factory called on missing key
d = defaultdict(int)           # default = 0 — use for counting
d = defaultdict(list)          # default = [] — use for grouping
d = defaultdict(set)           # default = set() — use for adjacency with no duplicates
d = defaultdict(lambda: float('inf'))  # custom default

    key = tuple(sorted(word))   # ('a','e','t') for "eat", "tea", "ate"

2.4 Anagram Detection — Three Approaches, One Insight

Anagram problems are the canonical hash map exercise. By seeing three progressively better solutions to the same problem, you internalize the thought process: brute force → sort → hash. This progression is how you should approach every problem in interviews.

python valid anagram (LC #242) — 3 approaches 
# ── APPROACH 1: Sort Both Strings — O(n log n) ───────────────────
def is_anagram_sort(s, t):
    if len(s) != len(t):
        return False
    return sorted(s) == sorted(t)
# Time: O(n log n), Space: O(n) for sorted copies
# Simple, readable — good starting point to mention, then optimize

# ── APPROACH 2: Counter Comparison — O(n) ────────────────────────
from collections import Counter
def is_anagram_counter(s, t):
    if len(s) != len(t):
        return False
    return Counter(s) == Counter(t)
# Time: O(n), Space: O(k) where k = unique characters (26 for lowercase alpha)
# Clean and Pythonic — good for interviews

# ── APPROACH 3: Single Array Count — O(n), no imports ────────────
def is_anagram_array(s, t):
    if len(s) != len(t):
        return False
    
    count = [0] * 26          # one slot per letter a-z
    for i in range(len(s)):
        count[ord(s[i]) - ord('a')] += 1   # increment for s
        count[ord(t[i]) - ord('a')] -= 1   # decrement for t
    
    return all(c == 0 for c in count)
# Time: O(n), Space: O(1) — technically O(26) = O(1)
# If ALL counts are 0, s and t have exact same character frequencies

# ── WHICH TO USE? ────────────────────────────────────────────────
# In interview: mention Approach 1 as brute force, code Approach 2 (clean)
# If interviewer asks "can you do better?", discuss Approach 3
# If problem says "Unicode strings" → must use dict, not 26-element array
💡 The Pattern That Generalizes

The key insight from anagram detection: two strings are anagrams if and only if their character frequency maps are identical. This insight generalizes: whenever you need to compare two things for "same composition regardless of order," think frequency map. This exact pattern appears in Minimum Window Substring, Permutation in String, and Ransom Note.

3.1 The Two Sum Pattern — Trading Space for Time

🧠 The Core Insight

The Two Sum problem asks: find two numbers in an array that add up to a target. Brute force is O(n²) — try all pairs. The hash map insight: for each number x, if target - x was seen before, we found our pair. We traded O(n) space for O(n) time improvement.

❌ Brute Force — O(n²)
def two_sum(nums, target):
  for i in range(len(nums)):
    for j in range(i+1, len(nums)):
      if nums[i]+nums[j] == target:
        return [i, j]
  return []
# Time: O(n²), Space: O(1)
# Fails for n=10,000: 100 million comparisons
✅ Hash Map — O(n)
def two_sum(nums, target):
  seen = {}  # val → index
  for i, num in enumerate(nums):
    complement = target - num
    if complement in seen:
      return [seen[complement], i]
    seen[num] = i
  return []
# Time: O(n), Space: O(n)
# One pass — as we go, check if complement exists
python two sum variations — the full pattern family 
# ── VARIATION: Two Sum with sorted array → Two Pointers ──────────
# If input is sorted, two pointers is O(n) and O(1) space (even better!)
def two_sum_sorted(nums, target):
    left, right = 0, len(nums) - 1
    while left < right:
        total = nums[left] + nums[right]
        if total == target:
            return [left, right]
        elif total < target:
            left += 1     # need bigger sum → move left pointer right
        else:
            right -= 1    # need smaller sum → move right pointer left
    return []

# ── VARIATION: Check if any two elements sum to target ───────────
def has_two_sum(nums, target):
    seen = set()       # just need existence, not index → use set
    for num in nums:
        if target - num in seen:
            return True
        seen.add(num)
    return False

# ── VARIATION: All pairs that sum to target ──────────────────────
def all_two_sums(nums, target):
    freq = Counter(nums)
    seen_pairs = set()
    result = []
    for num in nums:
        complement = target - num
        if complement in freq:
            pair = tuple(sorted([num, complement]))
            if pair not in seen_pairs:
                if num != complement or freq[num] >= 2:
                    result.append([num, complement])
                    seen_pairs.add(pair)
    return result
✅ The Pattern Template to Memorize

Template: seen = {} → iterate → for each element, compute complement = target - element → check if complement in seen → add current to seen. This one template, with minor variations, solves Two Sum, Three Sum (reduced to Two Sum), Four Sum, etc.

3.2 Frequency Map Pattern — Count Then Decide

The frequency map pattern: first count occurrences of all elements, then use those counts to answer the question. This is behind Top K Frequent Elements, Valid Anagram, First Unique Character, and more.

python top K frequent elements — the pattern in action (LC #347) 
# ── APPROACH 1: Sort by frequency — O(n log n) ───────────────────
def top_k_frequent_sort(nums, k):
    freq = Counter(nums)
    # Sort items by count descending, take first k
    return [num for num, _ in freq.most_common(k)]

# ── APPROACH 2: Bucket Sort — O(n) ───────────────────────────────
# Insight: frequency can be at most n (if all elements same)
# Create n+1 buckets where bucket[i] = all nums with frequency i
def top_k_frequent(nums, k):
    freq = Counter(nums)
    
    # bucket[i] contains all nums that appear exactly i times
    bucket = [[] for _ in range(len(nums) + 1)]
    for num, count in freq.items():
        bucket[count].append(num)
    
    # Collect from highest frequency buckets first
    result = []
    for i in range(len(bucket) - 1, -1, -1):
        for num in bucket[i]:
            result.append(num)
            if len(result) == k:
                return result

# Example: nums = [1,1,1,2,2,3], k = 2 → [1, 2]
# freq = {1:3, 2:2, 3:1}
# bucket = [[], [3], [2], [1], [], [], []]
# Scan from right: bucket[3]=[1] → result=[1]
#                  bucket[2]=[2] → result=[1,2] → len == k, return!

# Time: O(n), Space: O(n)
# The bucket sort trick avoids the O(n log n) sort entirely
Approach 1 O(n log n)
Approach 2 O(n)
SPACE O(n) both

3.3 Group By Pattern — Hash Map of Lists

Grouping problems: you want to collect elements that share some property into groups. The template is always: compute a hashable key for each element, use defaultdict(list) to group by that key.

python group anagrams (LC #49) — the canonical grouping problem 
# Group Anagrams — LC #49
# Input: ["eat","tea","tan","ate","nat","bat"]
# Output: [["bat"],["nat","tan"],["ate","eat","tea"]]

# ── APPROACH 1: Sort each word as key — O(n * k log k) ───────────
def group_anagrams_sort(strs):
    groups = defaultdict(list)
    for word in strs:
        key = tuple(sorted(word))   # must be tuple (list is not hashable)
        groups[key].append(word)
    return list(groups.values())

# ── APPROACH 2: Character count as key — O(n * k) ────────────────
# Avoids the k log k sort — uses character frequency tuple as key
def group_anagrams_count(strs):
    groups = defaultdict(list)
    for word in strs:
        count = [0] * 26
        for c in word:
            count[ord(c) - ord('a')] += 1
        key = tuple(count)           # (2,0,0,...,1,...) — hashable!
        groups[key].append(word)
    return list(groups.values())

Q01 / 10
What is the time complexity of list.pop(0) in Python? Why is it different from list.pop()? What should you use instead for O(1) removal from the front?
list.pop(0) is O(n) because when you remove the first element, every other element must shift left by one position to fill the gap. list.pop() (from the end) is O(1) — no shifting needed, just reduce the length. For O(1) removal from the front (queue operations), use collections.deque and popleft() — which is O(1) because deque maintains pointers to both ends.
Q02 / 10
Why can't you use a list as a dictionary key? What can you use instead when you need a hashable collection?
Lists are mutable — they can change after creation. If a list were used as a dict key and then modified, its hash would change, breaking the hash map's invariant (you'd never find the value again). Only immutable objects are hashable. Use a tuple instead — it looks like a list but is immutable. This is why tuple(sorted(word)) is the canonical key for anagram grouping.
Q03 / 10
Explain how a hash map achieves O(1) lookup. What happens internally when you do d["key"]?
Python computes hash("key") — a fixed-size integer derived from the key's value. It then does hash("key") % array_size to get a bucket index. It jumps directly to that bucket and returns the stored value. No iteration — just math and a direct memory access. This is why it's O(1). In the case of a collision (two keys hashing to the same bucket), Python chains them — worst case becomes O(n) if everything collides, but Python's hash functions make this extremely unlikely in practice.
Q04 / 10
What is a prefix sum array? Given arr = [2, 4, 1, 5, 3], write the prefix array and compute the sum of elements from index 1 to 3.
A prefix sum array stores cumulative sums: prefix[i] = sum of arr[0..i-1]. With a sentinel 0 at the start:

arr = [2, 4, 1, 5, 3]
prefix = [0, 2, 6, 7, 12, 15]

Sum from index 1 to 3 (inclusive): prefix[4] - prefix[1] = 12 - 2 = 10 (= 4+1+5 ✓). Built once in O(n), each query is O(1).
Q05 / 10
Explain Kadane's algorithm in one sentence. What is the "decision" it makes at each step?
Kadane's algorithm finds the maximum sum contiguous subarray by, at each element, deciding: is it better to start a new subarray here (take just this element), or extend the existing one? This is: curr = max(num, curr + num). Track the global best as you go: best = max(best, curr). Time O(n), space O(1).
Q06 / 10
Write the Two Sum solution using a hash map. What is the key insight that makes it O(n) instead of O(n²)?
The key insight: for each element x, its complement is target - x. Instead of searching the entire array for the complement (O(n) per element → O(n²) total), store each element in a hash map as we go. Check if the complement already exists in the map — O(1) lookup. One pass, O(n) total.

Q07 / 10
What is the difference between sorted(arr) and arr.sort()? When would you use each?
arr.sort() sorts in-place — modifies the original list, returns None. sorted(arr) returns a new list — original is unchanged. Use arr.sort() when you don't need the original order. Use sorted(arr) when you need to preserve the original, or when sorting any iterable (not just lists — e.g. sorted(dict.items())). Both are O(n log n) Timsort.
Q08 / 10
What does Counter.most_common(k) return? What is its time complexity?
most_common(k) returns a list of the k most frequent elements as (element, count) tuples, sorted from most to least frequent. Example: Counter("aabbc").most_common(2) → [('a', 2), ('b', 2)]. Time complexity: O(n log k) using a heap internally — or O(n log n) if k is None (returns all). The key interview fact: use this instead of manually sorting the full frequency dict when you only need the top k.
Q09 / 10
What is amortized O(1) and which Python list operation has this property?
Amortized O(1) means that while individual operations might occasionally be expensive, the average cost per operation over many operations is O(1). list.append() has this property. Most appends are O(1) — just write to the next slot. But when the list is full, Python allocates a new, larger array (2x size) and copies all elements — that one operation is O(n). However, since it happens rarely and the list doubles each time, the amortized cost across all appends is O(1).
Q10 / 10
What is the template for the "group by" pattern? Give the Group Anagrams key computation using sorted characters.
Template: groups = defaultdict(list) → for each item, compute a hashable key → groups[key].append(item) → return list(groups.values()). The creative part is the key computation.

For anagrams: key = tuple(sorted(word)) — all anagrams of the same word will have the same sorted character tuple. E.g. "eat", "tea", "ate" all produce ('a', 'e', 't'). Must be tuple (not list) because tuples are hashable.
