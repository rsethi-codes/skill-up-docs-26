Day 1 Quick Recall

Before new material, fire the synapses from yesterday. Don't scroll back to Day 1 — answer from memory. That friction is what builds retention.

Structural Typing

TypeScript checks shape, not names. { name: string, age: number } satisfies { name: string } because it has everything required, plus more.

type vs interface

Only interface supports declaration merging. Only type can express unions, primitive aliases, conditional types. They're both fine for objects.

Discriminated Unions

A union with a shared literal discriminant field (kind, type, status). Switching on the discriminant narrows the full union automatically. Add never in default for exhaustive checking.

infer Keyword

Pattern-matches inside conditional types. T extends () => infer R ? R : never extracts the return type of any function. Powers ReturnType, Awaited, Parameters.

unknown vs any

any turns off type checking. unknown forces you to check before using. Always reach for unknown when you don't know the type — never any in app code.

Utility Types

Partial, Required, Readonly, Pick, Omit, Record, Exclude, Extract. Know each one cold and be able to implement from scratch using mapped/conditional types.

Topic 01

Async & Error Handling

Almost everything in real applications is async. And async code fails. The question is whether your failures are typed — giving you compile-time guarantees — or untyped — giving you runtime surprises. Today you get the patterns that make async TypeScript bulletproof.

1.1 Typing Async Functions & Promises

🧠 Core Mental Model

An async function always returns a Promise. The type annotation Promise<T> declares what the promise resolves to. Async functions can never return void in practice — they return Promise<void>. And when they throw, they return a rejected Promise. The type system only tracks the resolved type, not the rejected type — which is the problem we solve with the Result pattern.

typescript promise typing — the full picture 
// ─── Basic async function return types ──────────────────────────
async function fetchUser(id: string): Promise<User> {
  const res = await fetch(`/api/users/${id}`);
  return res.json(); // ← Problem: json() returns Promise
}

async function deleteUser(id: string): Promise<void> {
  await fetch(`/api/users/${id}`, { method: "DELETE" });
  // returns Promise — nothing to return
}

// ─── Typing concurrent operations ─────────────────────────────
async function fetchDashboard(userId: string) {
  // Promise.all — all must succeed; type is tuple of each Promise's type
  const [user, posts, notifications] = await Promise.all([
    fetchUser(userId),          // Promise
    fetchPosts(userId),         // Promise
    fetchNotifications(userId), // Promise
  ]);
  // user: User, posts: Post[], notifications: Notification[] — all correctly typed

  // Promise.allSettled — survives partial failures
  const results = await Promise.allSettled([
    fetchUser(userId),
    fetchPosts(userId),
  ]);
  // results: PromiseSettledResult[]
  // Each result is { status: "fulfilled", value: T } | { status: "rejected", reason: unknown }

  results.forEach(result => {
    if (result.status === "fulfilled") {
      console.log(result.value); // User | Post[]
    } else {
      console.error(result.reason); // unknown — must narrow
    }
  });
}

// ─── Async arrow functions ────────────────────────────────────
const getUser = async (id: string): Promise<User> => {
  /* ... */
};

// ─── Awaited utility type — unwrap Promise type ───────────
type UserResult = Awaited<ReturnType<typeof fetchUser>>;
// = User (Promise → User)
// Use this to extract a promise's type when you don't own the function signature

1.2 The Error Handling Problem — Why try/catch Fails TypeScript

Here's the critical insight most TypeScript developers miss: TypeScript cannot type what a function throws. The language has no mechanism for declaring "this function throws a NetworkError or a ValidationError." Every catch (e) block gives you unknown — intentionally.

The Fundamental Problem — Throws Are Invisible to the Type System

  function fetchUser(id: string): Promise<User> { ... }
                                         ↑
                                   TypeScript sees this
                                   
  But what about when it throws? TypeScript sees NOTHING.
  It could throw: NetworkError | TimeoutError | ValidationError | RateLimitError | anything
  
  The promise type says "if it succeeds, you get a User"
  But the error path is completely untyped.
  
  This means you need to handle two problems:
  
  ① fetch() / axios throws on network failure — but not on 4xx/5xx
  ② json() can throw on malformed JSON
  ③ YOUR code can throw on bad data
  ④ ALL of these manifest as untyped exceptions in catch blocks
  
  The fix: NEVER let errors become exceptions. RETURN them instead.
🚫 The Deadly Mistake — Returning "any" Errors

The common pattern of catch (e) { return { error: e.message } } is wrong in two ways: (1) e is unknown so e.message is a TypeScript error, and (2) it loses the error type entirely, making the caller unable to distinguish error cases. TypeScript 4.0+ correctly types catch parameters as unknown, not Error, because anything can be thrown.

1.3 The Result Pattern — Type Your Errors

The Result pattern (from Rust/Haskell) makes errors first-class typed values. Instead of throwing, you return a discriminated union of success or failure. The caller is forced by TypeScript to handle both cases.

fetchUser(id)

typescript result pattern — complete production implementation 
// ─── Step 1: Define the Result type ──────────────────────────────
type Result<T, E = Error> =
  | { ok: true;  value: T }
  | { ok: false; error: E };

// Convenience constructors
const ok = <T>(value: T): Result<T, never> => ({ ok: true, value });
const err = <E>(error: E): Result<never, E> => ({ ok: false, error });

// ─── Step 2: Define typed error variants ─────────────────────────
type ApiError =
  | { kind: "network";     message: string }
  | { kind: "not_found";   id: string }
  | { kind: "unauthorized"; message: string }
  | { kind: "rate_limit";  retryAfter: number }
  | { kind: "validation";  fields: Record<string, string> };

// ─── Step 3: Functions that return Result ─────────────────────────
async function fetchUser(id: string): Promise<Result<User, ApiError>> {
  try {
    const res = await fetch(`/api/users/${id}`);
    
    if (res.status === 404) {
      return err({ kind: "not_found", id });
    }
    if (res.status === 401) {
      return err({ kind: "unauthorized", message: "Token expired" });
    }
    if (res.status === 429) {
      const retryAfter = parseInt(res.headers.get("Retry-After") ?? "60");
      return err({ kind: "rate_limit", retryAfter });
    }
    if (!res.ok) {
      return err({ kind: "network", message: `HTTP ${res.status}` });
    }

    const data: unknown = await res.json();
    // Validate data here with Zod (see section 1.5)
    return ok(data as User); // ideally use Zod.parse instead of cast
    
  } catch (e: unknown) {
    // Only actual network failures reach here (no internet, CORS, etc.)
    const message = e instanceof Error ? e.message : "Unknown network error";
    return err({ kind: "network", message });
  }
}

// ─── Step 4: The caller must handle both paths ────────────────────
async function loadProfile(userId: string) {
  const result = await fetchUser(userId);

  if (!result.ok) {
    // result.error is ApiError — fully typed, switch on kind
    switch (result.error.kind) {
      case "not_found":
        return showNotFound(result.error.id); // id is typed string
      case "unauthorized":
        return redirectToLogin();
      case "rate_limit":
        return showRateLimitMessage(result.error.retryAfter); // retryAfter: number
      case "network":
        return showNetworkError(result.error.message);
    }
  }

  // result.value is User — TypeScript KNOWS it's ok here
  const user = result.value;
  displayUser(user);
}
💡 Why This Is Better Than try/catch

With try/catch, the caller has no idea what errors to expect — the error type is invisible. With Result, the error possibilities are documented in the type signature. Adding a new error case to ApiError forces every caller to handle it (if they use exhaustive checking). You get compile-time proof that no error case is silently swallowed.

1.4 Typed Fetch — A Generic API Wrapper

fetch returns Response, and response.json() returns Promise<any>. You need a wrapper that: (1) types the response, (2) handles HTTP errors as Results, (3) is generic so one function works for all endpoints.

typescript production-grade typed fetch wrapper 
// ─── Generic fetch wrapper ────────────────────────────────────────
type FetchError =
  | { kind: "network"; message: string }
  | { kind: "http"; status: number; body: unknown }
  | { kind: "parse"; message: string };

async function apiFetch<T>(
  url: string,
  options?: RequestInit
): Promise<Result<T, FetchError>> {
  try {
    const res = await fetch(url, {
      headers: { "Content-Type": "application/json", ...options?.headers },
      ...options,
    });

    if (!res.ok) {
      const body: unknown = await res.json().catch(() => null);
      return err({ kind: "http", status: res.status, body });
    }

    try {
      const data: T = await res.json();
      return ok(data);
    } catch {
      return err({ kind: "parse", message: "Invalid JSON response" });
    }
  } catch (e: unknown) {
    const message = e instanceof Error ? e.message : "Unknown error";
    return err({ kind: "network", message });
  }
}

// ─── Usage — caller tells TypeScript what to expect ───────────────
interface User { id: string; name: string; email: string }

const userResult = await apiFetch<User>("/api/users/123");
const postsResult = await apiFetch<Post[]>("/api/posts?userId=123");

// POST with body
const createResult = await apiFetch<User>("/api/users", {
  method: "POST",
  body: JSON.stringify({ name: "Raghav", email: "r@test.com" }),
});

1.5 Zod — Runtime Safety Meets TypeScript

Here's the gap apiFetch<T> doesn't close: the API could return anything and you cast it to T. That's still a lie — you're telling TypeScript "trust me, this is a User" without actually checking. Zod validates at runtime and infers the TypeScript type automatically. One schema, two jobs.

🎬 WATCH NOW — Zod — The TypeScript-first Schema Validation Library
typescript zod + result pattern — the complete stack 
import { z } from "zod";

// ─── Define schema (runtime validator) ──────────────────────────
const UserSchema = z.object({
  id:    z.string().uuid(),
  name:  z.string().min(1).max(100),
  email: z.string().email(),
  role:  z.enum(["admin", "user"]),
  createdAt: z.string().datetime(),
});

// ─── Infer TypeScript type FROM the schema — single source of truth
type User = z.infer<typeof UserSchema>;
// { id: string; name: string; email: string; role: "admin" | "user"; createdAt: string }
// You NEVER write this manually — Zod generates it from the schema!

// ─── Validated fetch — actually safe ─────────────────────────────
async function fetchUser(id: string): Promise<Result<User, FetchError | { kind: "invalid"; errors: z.ZodIssue[] }>> {
  const raw = await apiFetch<unknown>(`/api/users/${id}`);

  if (!raw.ok) return raw;

  const parsed = UserSchema.safeParse(raw.value);
  if (!parsed.success) {
    return err({ kind: "invalid", errors: parsed.error.issues });
  }

  return ok(parsed.data); // parsed.data is User — ACTUALLY validated
}

// ─── Zod for form / request body validation ───────────────────────
const CreateUserSchema = UserSchema.omit({ id: true, createdAt: true });
type CreateUserInput = z.infer<typeof CreateUserSchema>;

// Extend or transform schemas:
const AdminSchema = UserSchema.extend({ permissions: z.array(z.string()) });
const PublicUserSchema = UserSchema.pick({ id: true, name: true }); // like Omit/Pick but for schemas
✅ Why Zod Is On Your Roadmap

You'll use Zod in the SyncSpace project (Week 5-6). It validates Express request bodies, Socket.IO message payloads, and API responses. It replaces manual type guards entirely. Learning it now means you're ready to use it properly in the project — not fumbling with it during build time.

Topic 02

Modules & tsconfig

The reason "TypeScript works on my machine but not in production" is almost always a module system or tsconfig misconfiguration. Understanding this deeply is what separates developers who fight tooling from developers who own it.

2.1 ESM vs CommonJS — The Module War

ES Modules (ESM)

The JavaScript standard since ES2015. Native browser support. Static imports (analyzed at compile time).

  • import x from 'y'
  • export default x
  • export { x, y }
  • Top-level await supported
  • Tree-shakeable by bundlers
  • Used by Next.js, Vite, modern Node ≥ 14

CommonJS (CJS)

Node.js legacy format, still used in older packages. Dynamic requires (evaluated at runtime).

  • const x = require('y')
  • module.exports = x
  • exports.x = x
  • No top-level await
  • Not tree-shakeable natively
  • Default in Node without "type": "module"
typescript module system in TypeScript context 
// TypeScript compiles ESM syntax to CJS or ESM based on tsconfig "module" setting

// ─── Your TypeScript source (always write ESM syntax) ─────────────
import { createServer } from "http";
import express from "express";
export { createServer };
export default express;

// ─── When compiled to CJS (module: "CommonJS") ────────────────────
// "use strict";
// const http_1 = require("http");
// const express_1 = require("express");
// exports.createServer = http_1.createServer;

// ─── When compiled to ESM (module: "ESNext") ─────────────────────
// import { createServer } from "http";
// import express from "express";
// export { createServer };
// export default express;

// ─── The interop problem ──────────────────────────────────────────
// CJS: module.exports = { a, b }
// ESM importing CJS: import pkg from 'cjs-pkg'; → pkg.default is the module
// This is why some imports look like: import { default as x } from 'pkg';
// or need: import x = require('pkg'); in older TypeScript

// ─── esModuleInterop: true (always enable!) ───────────────────────
// Without: import React from 'react' FAILS (React is CJS)
// With: TypeScript adds a default interop wrapper, import React from 'react' works

2.2 Declaration Files (.d.ts) — TypeScript's X-ray Vision

When TypeScript compiles your code, it can output .d.ts files alongside the JavaScript. These declaration files contain only type information — no runtime code. They let other TypeScript consumers use your library with full type safety, even though they only execute the JS.

What Declaration Files Are and Where They Come From

  Your TypeScript source                  After compilation
  ───────────────────                     ─────────────────
  
  src/utils.ts          →  dist/utils.js      (runtime code)
                        →  dist/utils.d.ts    (type declarations only)
  
  
  What dist/utils.d.ts looks like:
  
  ┌─────────────────────────────────────────────────────────────┐
  │  // No implementation code — just shapes                    │
  │  export declare function formatDate(d: Date): string;       │
  │  export declare type DateFormat = "ISO" | "US" | "EU";      │
  │  export declare interface Config { format: DateFormat; }    │
  └─────────────────────────────────────────────────────────────┘
  
  
  When another developer imports your package:
  
  import { formatDate } from "your-package";
          ↑ TypeScript reads the .d.ts file, not the .ts source
            It knows formatDate takes a Date and returns string
            Without ever seeing your implementation
typescript creating and using declaration files 
// ─── tsconfig to generate .d.ts ──────────────────────────────────
// tsconfig.json:
// "declaration": true  ← generates .d.ts files
// "declarationMap": true  ← generates .d.ts.map for Go To Source
// "declarationDir": "./dist/types"  ← where .d.ts files go

// ─── Writing a .d.ts manually (for JS libraries without types) ───
// File: my-js-lib.d.ts
declare module "my-js-lib" {
  export function doSomething(x: string): number;
  export interface Config { verbose: boolean }
  export default function create(config: Config): void;
}

// ─── Augmenting built-in types ───────────────────────────────────
// File: global.d.ts
declare global {
  interface Window {
    analytics: { track(event: string, props?: Record<string, unknown>): void };
    __APP_CONFIG__: { apiUrl: string; version: string };
  }
}
export {}; // make this a module (needed for declare global to work)

// Now window.analytics.track("page_view") is fully typed everywhere

2.3 @types Packages & DefinitelyTyped

Most popular JavaScript packages were written before TypeScript existed. The community maintains type definitions separately at DefinitelyTyped — a massive GitHub repository with .d.ts files for thousands of packages. These are published under the @types/ namespace on npm.

🔗 How @types Packages Work

When you run npm install express, you get the JavaScript. TypeScript has no idea what express exports. When you run npm install -D @types/express, you get node_modules/@types/express/index.d.ts — pure type declarations. TypeScript automatically picks these up from node_modules/@types/. The -D is important — they're dev-only, never shipped to production.

npm install -D @types/cors      # cors middleware types
npm install -D @types/bcrypt    # bcrypt hashing functions
npm install -D @types/jsonwebtoken  # JWT sign/verify

⚠️ moduleResolution: "bundler" vs "node" vs "NodeNext"

"node" is the old behavior — resolves like Node.js pre-ESM. "NodeNext" supports ESM in Node.js — requires you to write import './file.js' (with .js extension) even for .ts files. "bundler" is for Vite/Next.js/webpack — bundler handles resolution, TypeScript doesn't enforce file extensions. For Next.js always use "bundler". For a standalone Node.js backend use "NodeNext".

      "@hooks/*":       ["src/hooks/*"],
      "@lib/*":         ["src/lib/*"],
      "@types/*":       ["src/types/*"],
      "@server/*":      ["src/server/*"]
    }
  }
}

// ─── For Next.js: next.config.js (handles webpack + TS) ──────────
const nextConfig = {
  webpack: (config) => {
    config.resolve.alias = {
      ...config.resolve.alias,
      "@": path.resolve(__dirname, "src"),
    };
    return config;
  },

// OR use: tsc + tsc-alias for compiled output
// package.json:

💡 The Trap: tsconfig paths ≠ runtime aliases

TypeScript's paths only affects the TypeScript compiler — it tells TS where to look for types. It does NOT modify the compiled JavaScript. If your Node.js runtime sees @/lib/api, it has no idea what that means — it's not a real file path. You need tsconfig-paths (for ts-node) or tsc-alias / module-alias (for compiled output) to make aliases work at runtime.

Topic 03

TypeScript + Express

Express was written in JavaScript and its types are community-maintained. That means there are gaps you need to fill — especially around the Request object, which has no idea about your authenticated user or parsed body without help.

3.1 Extending Express's Request — Module Augmentation

You have a JWT auth middleware that decodes a token and attaches req.user. Express's type for Request has no user property. TypeScript errors. The solution is module augmentation — you extend Express's interface without modifying its source.

typescript extending Express Request — the correct pattern 
// ─── File: src/types/express.d.ts ─────────────────────────────────
// This file augments the Express module — adds properties to Request

import "express"; // IMPORTANT: must import to make this a module augmentation

declare module "express-serve-static-core" {
  interface Request {
    user?: {
      id: string;
      email: string;
      role: "admin" | "user";
    };
    requestId?: string; // for request tracing
  }
}

// ─── File: src/middleware/auth.ts ─────────────────────────────────
import { Request, Response, NextFunction } from "express";
import jwt from "jsonwebtoken";

export function requireAuth(req: Request, res: Response, next: NextFunction) {
  const authHeader = req.headers.authorization;
  
  if (!authHeader?.startsWith("Bearer ")) {
    res.status(401).json({ error: "No token" });
    return;
  }

  try {
    const token = authHeader.split(" ")[1];
    const payload = jwt.verify(token, process.env.JWT_SECRET!);
    // payload is string | JwtPayload — must narrow:
    if (typeof payload !== "object" || !payload || !("id" in payload)) {
      res.status(401).json({ error: "Invalid token" });
      return;
    }
    // TypeScript now knows req.user is available (from our augmentation)
    req.user = { 
      id: payload.id as string, 
      email: payload.email as string, 
      role: payload.role as "admin" | "user"
    };
    next();
  } catch {
    res.status(401).json({ error: "Invalid token" });
  }
}

// ─── In route handlers — req.user is now typed ────────────────────
router.get("/profile", requireAuth, (req, res) => {
  // req.user is { id: string; email: string; role: "admin" | "user" } | undefined
  if (!req.user) return res.status(401).send();
  res.json({ userId: req.user.id }); // ✅ fully typed
});

3.2 Middleware Typing — RequestHandler Generic

RequestHandler is Express's generic type for middleware and route handlers. It takes 4 type parameters that let you type the request and response bodies precisely.

typescript typed route handlers and middleware 
import { RequestHandler } from "express";

// RequestHandler
//   Params  = route params   (req.params)
//   ResBody = response body  (res.json(...))
//   ReqBody = request body   (req.body)
//   Query   = query params   (req.query)

// ─── Typed route params ───────────────────────────────────────────
type UserParams = { id: string };
type UserResponse = { user: User } | { error: string };

const getUser: RequestHandler<UserParams, UserResponse> = async (req, res) => {
  const { id } = req.params; // id: string — typed
  const user = await findUser(id);
  if (!user) {
    res.status(404).json({ error: "Not found" }); // must match UserResponse
    return;
  }
  res.json({ user }); // must match UserResponse
};

// ─── Typed request body ───────────────────────────────────────────
type CreateUserBody = { name: string; email: string; role: "admin" | "user" };
type CreateUserResponse = { user: User } | { error: string; fields?: Record<string, string> };

const createUser: RequestHandler<Record<never, never>, CreateUserResponse, CreateUserBody> = async (req, res) => {
  const { name, email, role } = req.body; // all typed!
  
  // Validate with Zod:
  const parsed = CreateUserSchema.safeParse(req.body);
  if (!parsed.success) {
    const fields = Object.fromEntries(
      parsed.error.issues.map(i => [i.path.join("."), i.message])
    );
    res.status(400).json({ error: "Validation failed", fields });
    return;
  }
  // ... create user, return result
};

3.3 Typed Environment Variables — Never as string Again

Every Node.js app uses process.env. The type of every value is string | undefined. Doing process.env.PORT as string everywhere is error-prone and hides missing variables until runtime. The right pattern validates all env vars at startup and exposes a typed env object.

typescript typed env variables — production pattern with Zod 
// ─── File: src/config/env.ts ──────────────────────────────────────
import { z } from "zod";

const EnvSchema = z.object({
  // Server
  NODE_ENV:    z.enum(["development", "test", "production"]).default("development"),
  PORT:        z.string().regex(/^\d+$/).default("3000").transform(Number),
  
  // Database
  DATABASE_URL: z.string().url(),
  DB_POOL_SIZE: z.string().default("10").transform(Number),
  
  // Auth
  JWT_SECRET:  z.string().min(32, "JWT_SECRET must be at least 32 characters"),
  JWT_EXPIRY:  z.string().default("7d"),
  
  // Redis
  REDIS_URL:   z.string().url(),
  
  // AWS
  AWS_REGION:         z.string().default("ap-south-1"),
  AWS_S3_BUCKET:      z.string(),
  AWS_ACCESS_KEY_ID:  z.string(),
  AWS_SECRET_ACCESS_KEY: z.string(),
});

// Validate on module load — app crashes immediately if env is misconfigured
const parsed = EnvSchema.safeParse(process.env);

if (!parsed.success) {
  console.error("❌ Invalid environment variables:");
  console.error(parsed.error.flatten().fieldErrors);
  process.exit(1); // FAIL FAST — don't start a broken server
}

export const env = parsed.data;
// env.PORT is number (not string!) — transformed by Zod
// env.NODE_ENV is "development" | "test" | "production" — not just string
// env.JWT_SECRET is string — guaranteed to exist

// ─── Usage throughout the app ─────────────────────────────────────
import { env } from "@/config/env";

app.listen(env.PORT, () => {          // number, not string
  console.log(`Server on port ${env.PORT}`);
});

const token = jwt.sign(payload, env.JWT_SECRET, { // string, guaranteed
  expiresIn: env.JWT_EXPIRY
});

3.4 The satisfies Operator — TypeScript 4.9

The satisfies operator is the answer to a specific frustrating problem: you want TypeScript to check that a value matches a type, but you want to keep the inferred literal type rather than widening to the annotation.

❌ Type Annotation — Widens Types
type Config = {
  theme: "light" | "dark";
  colors: Record<string, string>;
};

const config: Config = {
  theme: "dark",
  colors: {
    primary: "#007bff",
    secondary: "#6c757d",
  }
};

// Problem: TypeScript knows colors is Record
// NOT { primary: string; secondary: string }
// So this ERRORS:
config.colors.primary; // works but no autocomplete on keys
// TypeScript sees string, not "primary" | "secondary"
✅ satisfies — Checks Without Widening
type Config = {
  theme: "light" | "dark";
  colors: Record<string, string>;
};

const config = {
  theme: "dark",
  colors: {
    primary: "#007bff",
    secondary: "#6c757d",
  }
} satisfies Config;

// satisfies checks the shape matches Config ✅
// BUT infers the type as the literal object!
// So TypeScript knows config.colors has "primary" and "secondary"
config.colors.primary; // ✅ full autocomplete on keys
config.theme; // "dark" (not "light" | "dark")
typescript satisfies — real use cases 
// ─── Use case 1: Route configuration with full inference ──────────
type Route = { path: string; method: "GET" | "POST" | "PUT" | "DELETE" };

const routes = {
  getUser:    { path: "/users/:id", method: "GET"    },
  createUser: { path: "/users",     method: "POST"   },
  updateUser: { path: "/users/:id", method: "PUT"    },
  deleteUser: { path: "/users/:id", method: "DELETE" },
} satisfies Record<string, Route>;

// TypeScript checks all values are valid Routes ✅
// AND knows the exact keys ("getUser" | "createUser" | ...)
// AND knows routes.getUser.method is "GET" (not "GET" | "POST" | "PUT" | "DELETE")

// ─── Use case 2: Theme palette with specific colors ───────────────
const palette = {
  red:   ["#fde8e8", "#f87171", "#dc2626"],
  blue:  ["#dbeafe", "#60a5fa", "#2563eb"],
  green: ["#dcfce7", "#4ade80", "#16a34a"],
} satisfies Record<string, string[]>;

palette.red[0].toUpperCase(); // ✅ TypeScript knows red[0] is string, not string | undefined
// (without satisfies, Record would make indexing return string[] not string)

10 Questions — No Peeking

These are verbatim interview questions from real interviews at Indian product companies. Attempt each before revealing. Time yourself — you should be able to answer any of these in under 90 seconds verbally.

Q01
Write a Result<T,E> type and a function fetchUser that returns Result<User, ApiError>.
+
The Result type:
type Result<T, E = Error> = | { ok: true; value: T } | { ok: false; error: E }; type ApiError = | { kind: "not_found"; id: string } | { kind: "unauthorized"; message: string } | { kind: "network"; message: string }; async function fetchUser(id: string): Promise<Result<User, ApiError>> { try { const res = await fetch(`/api/users/${id}`); if (res.status === 404) return { ok: false, error: { kind: "not_found", id } }; if (res.status === 401) return { ok: false, error: { kind: "unauthorized", message: "Expired" } }; const user: User = await res.json(); return { ok: true, value: user }; } catch (e: unknown) { const message = e instanceof Error ? e.message : "Unknown"; return { ok: false, error: { kind: "network", message } }; } }
The key insight: the caller is forced to check result.ok before accessing result.value. TypeScript narrows the discriminated union automatically. This is zero-cost — no runtime overhead, just type safety.
Q02
How do you extend Express's Request interface to include a user property?
+
Use module augmentation — extend the Express interface by reopening it in a .d.ts file. The critical detail: augment "express-serve-static-core" (not "express" directly) because that's where Express's Request interface actually lives.
// src/types/express.d.ts import "express"; // must import to make this a module augmentation declare module "express-serve-static-core" { interface Request { user?: { id: string; role: "admin" | "user" }; } }
Without the import "express" line, this file is an ambient declaration and the augmentation applies globally but differently. With the import, TypeScript treats it as a module augmentation scoped correctly.
Q03
What is the difference between declaration merging and module augmentation?
+
Declaration merging happens when you declare the same interface name twice in the same scope. TypeScript merges them into one combined interface. It only works for interfaces, not type aliases.

Module augmentation is declaration merging applied to another module's exported types. You use declare module "x" { interface Y { ... } } inside a file that imports from the module. This extends the module's types without modifying its source.

Module augmentation is how the Express Request extension works, how @types/jest adds describe/it to global scope, and how window.analytics gets typed globally.
Q04
Explain tsconfig strict mode. What does it enable and why should you always use it?
+
"strict": true is a shorthand that enables a group of strict flags simultaneously:

• noImplicitAny — variables can't accidentally become any through inference
• strictNullChecks — null and undefined are distinct types, not assignable to everything
• strictFunctionTypes — function parameter types are checked contravariantly
• strictPropertyInitialization — class properties must be initialized in the constructor
• useUnknownInCatchVariables — catch block e is unknown not any

Without strict mode, TypeScript is dramatically less useful. null bugs slip through, accidental any types spread silently, and catch blocks give you false confidence. Always use strict: true. There's no valid reason to skip it in new projects.
Q05
How do you type environment variables safely in TypeScript without using process.env.FOO as string everywhere?
+
Validate all env vars at startup using Zod and export a typed env object:
const EnvSchema = z.object({ PORT: z.string().transform(Number), DATABASE_URL: z.string().url(), JWT_SECRET: z.string().min(32), NODE_ENV: z.enum(["development", "production"]), }); const result = EnvSchema.safeParse(process.env); if (!result.success) { console.error(result.error); process.exit(1); } export const env = result.data; // typed, validated, never undefined
Benefits: (1) App fails immediately at startup if any variable is missing or malformed. (2) env.PORT is number not string. (3) env.NODE_ENV is the union literal, not string. (4) No type assertions anywhere — the types are earned, not declared.
Q06
What is a discriminated union and how does it improve error handling over try/catch?
+
A discriminated union is a union type where every member has a shared literal field (the discriminant). Switching on this field narrows the type to the specific member.

In error handling: with try/catch, the error type is unknown — no one knows what errors a function throws. With a Result discriminated union ({ ok: true, value: T } | { ok: false, error: E }), the error possibilities are part of the function's type signature. The caller must check result.ok before accessing the value. New error types added to the union become compiler errors at every unhandled call site. This is compile-time exhaustive error handling — something try/catch can never provide.
Q07
Write a typed generic fetch wrapper: async function apiFetch<T>(url: string): Promise<T>
+
async function apiFetch<T>(url: string, options?: RequestInit): Promise<T> { const res = await fetch(url, { headers: { 'Content-Type': 'application/json', ...options?.headers }, ...options, }); if (!res.ok) { throw new Error(`HTTP error: ${res.status}`); } return res.json() as Promise<T>; } // Usage: const user = await apiFetch<User>("/api/users/1"); // user: User
Note: This is the simpler version that throws. For production, upgrade to return Result<T, FetchError> and never throw. The T is the caller's promise about what the API returns — if the API lies, the cast is wrong but that's a runtime problem Zod solves.
Q08
What does moduleResolution: "bundler" do in tsconfig vs "node"?
+
"node" (legacy) — TypeScript resolves imports like Node.js CommonJS: tries x.ts, x/index.ts, etc. Requires no file extensions in imports. Used for older projects.

"bundler" (new, for Next.js/Vite/webpack) — TypeScript defers to the bundler for actual resolution. Allows imports without file extensions, supports exports field in package.json. Designed for modern build tools that handle module resolution themselves.

"NodeNext" — strict ESM mode for modern Node.js. Requires explicit .js extensions in imports (even for .ts files), and respects type: "module" in package.json. Use this for Node.js-only backends targeting Node 18+.

Practical rule: Next.js = "bundler". Pure Node backend = "NodeNext". Old code = "node".
Q09
How do you create path aliases in TypeScript and what problem do they solve?
+
In tsconfig.json:
{ "compilerOptions": { "baseUrl": ".", "paths": { "@/*": ["src/*"] } } }
This lets you write import { Button } from "@/components/Button" instead of "../../../components/Button".

Problems solved: (1) Fragile imports — relative imports break when files move. Aliases are absolute from project root. (2) Readability — you can tell immediately whether an import is from your code or a package. (3) Refactoring — move a file anywhere; imports don't change.

Critical gotcha: tsconfig paths only affects TypeScript. Runtime (Node.js) doesn't know about them — you need tsconfig-paths for ts-node, or tsc-alias for compiled output.
Q10
What is the purpose of the satisfies operator introduced in TypeScript 4.9?
+
satisfies validates that a value matches a type without widening the inferred type to the annotation.

Problem it solves: If you write const x: Record<string, string[]> = { red: ["#fff"] }, TypeScript knows x has string keys with string[] values, but not that it has a specific "red" key. With const x = { red: ["#fff"] } satisfies Record<string, string[]>, TypeScript checks the shape AND retains knowledge that x.red specifically exists.

Best use cases: configuration objects, route maps, theme definitions — anywhere you want type validation but also want to keep the specific literal inference for autocomplete and safety.
